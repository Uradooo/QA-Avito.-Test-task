# Тест-кейсы для проверки API

### API_TC_001: Создание объявления с типовыми данными (POST /api/1/item)

**Приоритет**: Высокий

**Цель**: Проверить, что API успешно принимает корректный POST-запрос для создания объявления и возвращает статус 200, а затем GET-запрос возвращает данные, совпадающие с исходными.

**Шаги**:

1. Отправить POST-запрос по адресу `/api/1/item` с телом:

```json
{
  "sellerId": 111112,
  "name": "Первый тест",
  "price": 1000,
  "statistics": {
    "contacts": 10,
    "likes": 10,
    "viewCount": 100
  }
}
```
2. Убедиться, что ответ имеет статус 200 и содержит поле `status` со строкой вида:

```
"Сохранили объявление - <id>"
```
3. Извлечь идентификатор объявления из ответа.

4. Выполнить GET-запрос по адресу `/api/1/item/{id}`.

5. Сравнить полученные поля (`sellerId`, `name`, `price`, `statistics`) с данными, отправленными в POST-запросе.

**Ожидаемый результат:**

* Статус ответа 200.
* Данные, возвращаемые GET-запросом, полностью совпадают с отправленными

---

### API_TC_004: Проверка валидации поля `sellerId` с пустым значением (POST /api/1/item)

**Приоритет**: Высокий

**Цель**: Убедиться, что API корректно отрабатывает с некорректным запросом для `sellerID`

**Шаги**:

1. Отправить POST-запрос по адресу `/api/1/item` с телом:

```json
{
  "sellerId": null,
  "name": "Первый тест",
  "price": 1000,
  "statistics": {
    "contacts": 10,
    "likes": 10,
    "viewCount": 100
  }
}
```

**Ожидаемый результат:**

* Статус ответа 400 Bad Request.

**Также могут быть протестированы следующие запросы:**

* Отправить POST-запрос, где `sellerID` задан с буквенными символами (например, "abc123").
* Отправить POST-запрос, где `sellerID` имеет слишком большое значение (превышает 999999, например 1000000)
* Отправить POST-запрос, где `sellerID` имеет слишком маленькое значение (меньше 111111, например 111110)
* Отправить POST-запрос, где `sellerID` равен отрицательному значению (например, -1)
